Frogs And Freedom

Encounter System

1. Although the collision presets are set to ignore everything except the player, an interface that only the player has is used to confirm the overlapping actor is the player.

The EncounterData property is a data asset that contains the enemy type, quantity and event trigger for each wave of the encounter.

2. Because designers were constantly iterating on the number of patrol routes and spawn locations, I used EQS instead of manually require designers to update arrays.

PopulatArraysInPatrolRouteManager calls RunEqsFindPatrolRoutes() which lives on another component. RunEqsFindActorSpawners lives on this component.

CanWaveSpawn and HasWaveSpawned are properties in the DataAsset not exposed to designers that are updated during gameplay. The for loop uses a reference of the wave data for each wave to reset the values to their default of false.

3. Designers assign gameplay tags 'wave.#' to the children of the ActorSpawner. Each element in the dataAsset's array has a 'wave' tag. The CurrentlyActiveWave is the tag of current wave.

The CurrentlyActiveEncounterTrigger is a box that encloses the entire encounter space. The EQS query finds all the actors of the object ype 'ActorSpawner' within the bounds of the encounter.

If the returned actors have a tag that matches the current wave, then they are added to the array of possible spawn locations.

4. This EQS takes in the EncounterTrigger as the context. The dimensions of the trigger component are used to set the bounds for a Box Multi Trace.

ObjectTypesToSearchFor is an instance editable array that is exposed to the EQS. In this example, the object type passed in is ActorSpawner.

Each ActorSpawner in the encounter is added to the AddGeneratedActors function, which is the array on actors the EQS returns.

5. For each encounter, could add up to ten waves to the array of a DA_Encounter. Each encounter trigger had one of these assets.

Quantity Spawned in Wave: total number of a given enemy type to spawn

Batch size: number of actors to spawn at a single spawn location (ActorSpawner class). If the Quanity=6 and the Batch-size = 2, then 3 batches, each with 2 enemies, would spawn.

If there were 3 ActorSpawners with a match wave.# gameplaytag, the 3 batches would all spawn at different locations. If there were fewer than 3, the array of possible spawn locations loops back to index 0.

Wave Trigger: designers could chose between Encounter-Start, Objective or Enemy-Population as s means to trigger a wave.

6. A unit test was used to automate the process of ensuring the correct data was entered.

When a new EncounterTrigger child was created, it couldn't be save until a DA_EncounterData was assigned to it's member variable.

When a new DA_EncounterData was created, it could be saved until the gameplay tags and values were properly set.

AI Perecption

1. Get the TeamID of the possessed Character. The TeamID uses the IGenericTeamAgentInterface, with overridden functions shown in the next slide.

Bind to the OnTargetPerceptionUpdated delegate, which is fired when a stimulus is perceived.

Bind to the delegate OnTargetPerceptionForgotten, which is fired when the stimulus age is greater than the maxAge property.

OnTargetPerceptionUpdated is only called if the property bForgetStaleActors is set to true. This is only accessible in the Default.ini

2. The IGenericTeamAgentInterface is implemented on the ACharacterBase class, which the enemy class and player class both inherit from. The AIController class already has this interface.

GetTeamAttitudeTowards() is overriden to check if the AActor passed in from a perception source has the same TeamID as the ACharacterBase this controller possesses. It returns ETeamAttitude as either 'Hostile, Friendly or Neutral.'

3. The AIPerceptionComponent is confirured to have sight, hearing, and damage senses. The ACharacterBasePlayer class is sensable using sight and damage sense, while AGeneratorBase is sensable over the hearing sense.

IsUpdatedSenseOfActorOfAGivenType() checks the type of stimulus and then calls the corresponding handle function for that sense.

4. EAISense is a enum defined in the AAIControllerMobileFrogBase class which tracks the type of sense/stimulus. However, Stimulus.Type is a FAISenseID so the locally defined EAISense can't be used to check the stimulus type.

GetSenseIDofSenseAsEnum() converts the local enum EAISense to the unreal stimulus type FAISenseID so it can be compared to the Stimulus parameter.

5. To avoid casting, the CharacterBasePlayer class implements the UPlayerCharacterable interface, which is used to check if an actor is the player.

HandleSenseSight(), changes the 'state' blackboard-key to Attacking. A check on the current state is doen to avoid unnecessary changes. HandleSenseHearing() and HandleSenseDamage() work similiarly but set a different 'state.'

GetAICurrentState(), UEnums in the blackboard are stored as int8, a static_cast is used to cast to the EAIStates type.

SetBBkeyTargetAndState(), when NewState is EAIStates::Attacking or EAIStates::Investigate, it takes in the sensed AActor as the AttackTarget. When NewState is EAIStates::Passive, AttackTarget is nullptr.

6. When the age of the AI's hearing or damage perception exceeds it's maxLife, it might still have sight perception. The player is the only AActor that can be perceived with the sight sense.

GetLastStimulus(), checks if the AI still has sight info on the player and if it doesn't, the 'state' is set to EAIState::Passive and the 'TargetObject' is set to nullptr. This function is shown on the next slide

The token system is gone over in more detail in the 'Token System' section below. The enemy character class, CharacterBaseOwner, implements the IAttackTokenRequesterable interface and the player character class implements the IAttackTokenGivererable interface.

7. The perception info on the player character is searched for and returned in the references bSensedSuccessfully and OutStimulus.

GetActorsPerception() takes a reference of the struct FActorPerceptionBlueprintInfo which holds perception info for a sensed AActor.

GetSenseIDofSenseAsEnum(), checks if the AI has sight perception info on the player character.

AI Behvaiour

1. To avoid casting, the interface IFrogAIable is used to call behaviour functions like attack and jump as well as bind and unbind events in Tasks.

The behaviour tree is event driven. Execution is controlled by bind to events and only continuing once a message is received. Here, GetPatrolRouteAndSetDestinationLocation() is called when the enemy has landed, ensuring execution stop when the AI is jumping.

Bind to the delegate OnTargetPerceptionForgotten, which is fired when the stimulus age is greater than the maxAge property.

OnTargetPerceptionUpdated is only called if the property bForgetStaleActors is set to true. This is only accessible in the Default.ini

2. GetPatrolRoute() returns a spline that is assigned to AI when it's pulled from the pool.

To avoid collisions, an EQS is run at the world location of the current index along the spine. The EQS returns an unoccupied location that it has line of sight to.

3. UFrogDestinationTest gather the 'destination' positions stored in each AI's blackboard, then scores every EQS candidate point by the average distance from that point to all the current positions of AI's. The score is the mean distance to every enemy AI's location.

The bb_key 'DestinationLocation' is stored as the local variable, DestinationLocationName. The Position array is populated with the vector value held within each AI's blackboard.

FEnvQueryInstance::ItemIterator loops over each EQS candidate item being tested. GetItemLocation(QueryInstance, It.GetIndex() gets the world location of the EQS item. For every element in the Positions array, the distance is calculated and added to the DistanceLengths array.

It.SetScore() sets the score the EQS candidate based on the average distance.

4. OnLandedFinishedExecute() is bound to the LandedDelegate, meaning the behaviour tree stays in this task until the AI lands.

JumpSpecialToDestination() is an interface function that calls a custom jump function, JumpToDestination().

JumpToDestination() finds the vector between the AI's current location and it's destination and launches the possessed character along that vector. The AI can jump to any location the EQS returns.

5. The task, BTT_RequestAttackToken first checks if the controlled Pawn already has claimed a token from the AttackTarget. If it hasn't, then it request one via the interface function RequestAttackToken().

See the 'Attack Token System' section for more details on the token system.

The behaviour tree shown here is the 'attack' state side of the tree. This slide shows the task BTT_RequestAttackToken and the next slide shows BTT_AttackFlusher.

6. AttackFinishedExecuteSuccessful() is bound to a delegate, AttackBehaviourFinishedDelegate, using a helper function on the FrogAIable interface, BindToAttackBehaviourFinishedDelegate().

Functions controlling behaviour such as IdealBehaviour(), InvestigateBehaviour() and AttackBehaviour(), and functions for binding and unbinding to delegates that broadcast when this behaviour finishes is all owned by the FrogAIable interface. The base enemy class, ACharacterBaseFrog, implements the FrogAIable interface.

AttackBehaviour() is called on the controlled pawn, which calls Attack() with the BP_CharacterFrogFlusher class, a child of ACharacterBaseFrog.

If the attack is successful, ReturnAttackTokens() is called and AttackBehaviourFinishedDelegate is broadcast.

7. The perception info on the player character is searched for and returned in the references bSensedSuccessfully and OutStimulus.

GetActorsPerception() takes a reference of the struct FActorPerceptionBlueprintInfo which holds perception info for a sensed AActor.

GetSenseIDofSenseAsEnum(), checks if the AI has sight perception info on the player character.
